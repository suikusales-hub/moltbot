01. Introduction

Welcome to MoltbotWiki, the definitive unofficial resource for architects, developers, and power users deploying Moltbot in production environments.

While the official guide covers the basics, MoltbotWiki dives deeper. This is where the community documents the edge cases, the optimization hacks, and the complex integration patterns that turn a standard Moltbot instance into a bespoke enterprise automation engine.

Moltbot represents a paradigm shift from static scripts to agentic workflows. Unlike traditional chatbots that simply respond to text, Moltbot observes, plans, and executes. This wiki serves as the map for that new territory, helping you navigate the complexities of autonomous AI deployment.

02. Getting Started: The "Right" Way

Installing Moltbot is deceptive simple, but configuring it for scalability requires foresight. Many users start with a simple Docker container but soon hit bottlenecks when scaling to hundreds of concurrent agent tasks.

Prerequisites Checklist

  • ✓ Docker Engine 24.0+ (with Buildx support)
  • ✓ Python 3.10+ (for local controller scripts)
  • ✓ Redis Stack (required for vector memory persistence)
  • ✓ 8GB+ RAM (16GB recommended for local LLM inference)
bash

# Clone the community extended repo

git clone https://github.com/moltbot-community/moltbot-extended.git

cd moltbot-extended

# Start with the production profile

docker-compose --profile production up -d

For a step-by-step walkthrough of the initial boot sequence, refer to our comprehensive tutorial. If you are setting this up for a corporate environment, ensure you review the privacy configuration guide first to prevent data leakage.

03. System Architecture

The Brain (Orchestrator)

The central decision-making unit. It parses user intent and routes tasks to specialized sub-agents. See Behind the Scenes for the logic flow.

Memory Matrix (Redis)

Moltbot's long-term memory. Unlike standard context windows, this vector database allows the agent to recall interactions from months ago.

Understanding the architecture is crucial when debugging. A common misconception is that Moltbot is a monolithic LLM wrapper. It is actually a microservices mesh. The `Agent Router` service determines whether a query needs the creative writing module or the code execution sandbox.

This modularity is what protects your infrastructure. If the LLM layer is compromised via prompt injection, the sandbox isolation prevents the agent from executing harmful system commands.

04. Security Protocols

Warning: Default configurations are optimized for ease of use, not maximum security. Before exposing Moltbot to the public internet, you must implement the "Ironclad" protocol.

The "Ironclad" Configuration

  1. Disable Recursive Tool Execution: Prevent the agent from calling tools that call other tools indefinitely. This mitigates the risk of Agentic Trojan Horses.
  2. Identity Verification: Implement strict OAuth2 scopes. Read about Hybrid Identity attacks to understand why API keys are not enough.
  3. Sandboxing: Run all code execution environments in ephemeral Docker containers with no network access, unless explicitly whitelisted.

Check the Active Threats Advisory for the latest CVEs affecting agentic frameworks.
